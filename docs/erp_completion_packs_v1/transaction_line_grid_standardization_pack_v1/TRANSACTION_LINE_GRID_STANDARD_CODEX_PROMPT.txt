Work only on branch main.

This is TRANSACTION-LINE-GRID-STANDARDIZATION-01.

This is a cross-cutting transaction-line correction pack.
Do NOT treat this as a cosmetic-only pass.
Do NOT fix only Quote.
Do NOT leave past transaction packs with weaker line-entry rules.
Do NOT use repeated large card-style line blocks for desktop transaction entry.

Mission:
Apply the universal desktop transaction line-grid standard across all transaction screens and completion packs.

Control pack:
docs/erp_completion_packs_v1/transaction_line_grid_standardization_pack_v1/

Required files:
- TRANSACTION_LINE_GRID_STANDARD_BENCHMARK_V1.xlsx
- TRANSACTION_LINE_GRID_STANDARD_COMPLETION_SPEC.md
- TRANSACTION_LINE_GRID_STANDARD_ADDENDUM.md

Hard rule:
Every desktop transaction screen that can contain multiple item/service/material/account lines must use a compact editable grid/table.
Card-per-line desktop transaction entry is a failure condition.

Phase 4 — Transaction tests
Create/update tests:

- QuoteLineGridFlow.test.tsx
- SalesOrderLineGridFlow.test.tsx
- PurchaseOrderLineGridFlow.test.tsx
- GoodsReceiptLineGridFlow.test.tsx
- InventoryMovementLineGridFlow.test.tsx
- ProductionReceiptLineGridFlow.test.tsx
- DispatchLineGridFlow.test.tsx

Tests must prove:
1. new transaction opens a header + editable line grid,
2. user can add at least 3 lines,
3. user can edit multiple line values,
4. user can remove the middle line,
5. save draft/save persists all remaining lines where backend exists,
6. reopened transaction shows all lines where backend exists,
7. totals recalculate from all lines where applicable,
8. screen does not render full card per line as the desktop entry pattern,
9. no first-line-only implementation remains.

If backend persistence is not available:
- test must prove save is disabled with reason,
- but UI must still not be a dead button or misleading card-based transaction form.
